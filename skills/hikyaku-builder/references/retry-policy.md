# リトライポリシー

## 原則

AIが問題解決を試みる際、**同じアプローチでの試行回数に上限を設ける**。
上限を超えた場合は手を止め、ユーザーに状況を報告して判断を仰ぐ。

消費したトークンは戻らない。闇雲なリトライは問題を解決せずトークンだけを消費するリスクがある。

## 試行回数の上限

| 問題の種類 | 上限 | 上限超過時の対応 |
|-----------|----|----------------|
| ビルドエラー | 2回 | 状況を報告し、ユーザーに判断を仰ぐ |
| テスト失敗 | 2回 | 状況を報告し、ユーザーに判断を仰ぐ |
| lint / format エラー | 2回 | 状況を報告し、ユーザーに判断を仰ぐ |
| 型エラー | 2回 | 状況を報告し、ユーザーに判断を仰ぐ |
| 外部API・環境起因のエラー | 1回 | 即座にユーザーに報告（自力解決の見込みが低い） |

※ 「回数」は **根本原因が同じ試行** を通算で数える。表面的なアプローチ変更（パラメータ変更・環境クリーン等）ではリセットしない。根本原因が異なると判断できる場合のみリセットする。判定基準は「同一原因の判定基準」を参照。

## 同一原因の判定基準

リトライカウントのリセットは「根本原因が変わった」場合にのみ行う。

### カウントをリセットしない（同一原因とみなす）例:
- コマンドのオプションやフラグを変えて再実行する
- 環境のクリーン（キャッシュ削除、再インストール等）をしてから再実行する
- ポート番号や出力先ディレクトリなどのパラメータを変えて再実行する
- デバッグオプションを追加して再実行する
- エラーの直接的な症状だけに対処して再実行する

これらは表面的な変更であり、根本原因が同じなら同一試行としてカウントする。

### カウントをリセットする（別原因とみなす）例:
- エラーメッセージが根本的に異なる（前回は依存解決エラー → 今回は型エラー）
- 修正対象のコード箇所が異なる
- 問題の原因仮説が変わった（前回はコード起因 → 今回は環境設定起因と特定）

## 報告フォーマット

上限に達した場合、以下の情報をユーザーに提示する:

```
## エラー報告

### 問題
（何が起きているか）

### 試したこと
1. （1回目のアプローチと結果）
2. （2回目のアプローチと結果）
3. （3回目のアプローチと結果）

### 考えられる原因
- （原因仮説A）
- （原因仮説B）

### 提案する次のアクション
- A: （アプローチ案）
- B: （別のアプローチ案）
- C: 一旦スキップして先に進む
```

## リトライ前に確認すべきこと

無駄な試行を防ぐため、リトライの前に以下を確認する:

1. **エラーメッセージを正確に読んだか** — 表面的な症状ではなく根本原因を特定する
2. **前回と同じアプローチになっていないか** — 同じことを繰り返しても結果は変わらない
3. **問題のスコープは限定できているか** — 影響範囲を最小化してから修正する
4. **自力で解決可能な問題か** — 環境・権限・外部サービスの問題はユーザーに委ねる
