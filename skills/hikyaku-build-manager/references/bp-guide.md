# ビルドポイント（BP）見積もりガイド

- AIエージェントとの1セッション（20万トークン）で 実装が完了することを基準とする。
- この基準はあくまでHikyakuワークフロー内で用いる定量的な数値である

## BP定義

| BP | 規模感 | セッション見通し |
|----|--------|---------------|
| **1** | 単一ファイルの修正・追加 | 余裕で完結。複数ビルドをまとめてもよい |
| **2** | 小さな機能追加 | 余裕で完結 |
| **3** | 標準的な機能単位 | 完結（レビュー修正込みで余裕あり） |
| **5** | やや大きめの機能単位 | 完結 |
| **8** | 大きな機能単位 | 完結（レビュー修正込みでギリギリ） |
| **13** | 非常に大きい | 確実に超過。**分割必須** |

**1セッション完結ライン: BP 5以下** — 分割による損失（コンテキスト分断、ビルド間の依存増加など）が大きい場合に限り BP 8 も許容する

## 見積もり手順

### Step 1: ワークスペースを特定する

ビルドが関わるワークスペース（パッケージ）を特定する。
単一ワークスペースの場合は Step 2 へ進む。

例:
- バックエンドのみ → 1ワークスペース
- フロントエンドのみ → 1ワークスペース
- backend + frontend → 2ワークスペース
- api-1 + api-2 + frontend → 3ワークスペース

### Step 2: ワークスペースごとにBPを算出する

各ワークスペースについて、主要指標と加算要素からBPを算出する。

#### 主要指標（ワークスペース内で最大値ベース）

| 指標 | BP1 | BP2 | BP3 | BP5 | BP8 | BP13 |
|------|-----|-----|-----|-----|-----|------|
| 新規ファイル数 | 1-2 | 3-5 | 6-15 | 16-20 | 21-30 | 30+ |
| 実装行数（テスト含む） | ~200 | ~500 | ~1,000 | ~2,000 | ~3,000 | 3,000+ |
| API操作数 | 0 | 1 | 2 | 3-4 | 5-7 | 8+ |
| 画面/ページ数 | 0 | 1 | 2 | 3 | 4-5 | 5+ |
| DBテーブル/エンティティ数 | 0-1 | 1 | 2-3 | 3-4 | 5-6 | 6+ |

**算出方法: ワークスペースに関連する指標でBPを判定し、最大値を採用する**

#### 加算要素（ワークスペースごとに適用）

| 要素 | 加算値 | 例 |
|------|--------|---|
| 基盤セットアップを含む | +1 | ORM導入、UIライブラリ初期化、CI構築、共通コンテキスト/ミドルウェアの構造変更 |
| 外部API連携を含む | +1 | 認証プロバイダ、決済API、オブジェクトストレージ（S3等） |
| 既存コードの大規模リファクタ | +1 | 新規より既存コード読み込みのコンテキスト消費が大きい |
| 影響ファイル数 4〜6 | +1 | 変更の影響を理解するために読み込むファイルが多い |
| 影響ファイル数 7〜10 | +3 | 共通インターフェースの変更、API レスポンス形式の変更 |
| 影響ファイル数 11〜 | +4 | DBスキーマ変更、広範囲に参照される型の変更 |

**影響ファイル数について:**
変更するファイルだけでなく、変更の影響を理解するために読み込む必要があるファイルを含む概算値。
codebase-survey.md や設計ドキュメントを参考に見積もる。正確でなくてよい — 見積もりが外れた場合は build-manager でビルドを分割する。

**ワークスペースBP = 主要指標の最大値 + 加算要素の合計**

### Step 3: 合計BPを算出する

全ワークスペースのBPを合計する。

```
合計BP = ワークスペース1のBP + ワークスペース2のBP + ...
```

合計BPが5を超える場合は分割を検討する。

## クイックリファレンス

### 単一ワークスペースの場合

```
□ 新規ファイル何個？       → 15以下ならBP3、20以下ならBP5
□ 実装何行くらい？         → 1,000以下ならBP3、2,000以下ならBP5
□ API操作いくつ？          → 2以下ならBP3、4以下ならBP5
□ 画面いくつ？             → 2以下ならBP3、3以下ならBP5
□ 基盤セットアップある？    → あれば+1
□ 外部API連携ある？        → あれば+1
□ 影響ファイルいくつ？      → 3以下なら+0、6以下なら+1、10以下なら+3、11以上なら+4

最大値 + 加算 = BP → 5以下ならOK、8は許容、13以上なら分割
```

### 複数ワークスペースの場合

```
□ ワークスペースをリストアップ
□ 各ワークスペースのBPを上記手順で算出
□ 合計する

合計BP → 5以下ならOK、8は許容、13以上なら分割
```

## BP 13以上の分割戦略

```
Step 1: 主要な分割軸を特定する
  ├─ 「基盤セットアップ」と「機能実装」が混在 → 分離
  ├─ 独立した画面が3つ以上 → 画面単位で分割
  ├─ 独立したドメインが3つ以上 → ドメイン単位で分割
  ├─ ワークスペースが分離可能 → ワークスペース単位で分割
  └─ 外部連携が2つ以上 → 連携先ごとに分割

Step 2: 分割後の各ビルドのBPを再算出

Step 3: 全ビルドがBP 8以下になるまで繰り返す
```

## BP別の具体例

### バックエンド

| BP | 例 | ベースBPの根拠 |
|----|---|---------------|
| 1 | バグ修正（1ファイル）、既存APIにフィールド追加 | ファイル1-2 |
| 2 | バリデーションルール追加+テスト | API 1, ファイル3-5 |
| 3 | 1リソースの参照API（一覧+詳細）+テスト | API 2, ファイル6-8 |
| 5 | 1ドメインのCRUD API+テスト | API 3-4, ファイル16前後 |
| 8 | 複数ドメインのAPI一式 | API 5-7 |
| 13 | マイクロサービス1つの新規構築 | API 8+, ファイル30+ |

### フロントエンド

| BP | 例 | ベースBPの根拠 |
|----|---|---------------|
| 1 | ボタン追加、スタイル修正 | ファイル1-2 |
| 2 | 既存画面に機能追加 | 画面1, ファイル3-5 |
| 3 | 1フォーム画面+バリデーション | 画面2, ファイル6-10 |
| 5 | 2-3画面+共通コンポーネント | 画面3 |
| 8 | 4画面以上+共通コンポーネント群 | 画面4-5 |
| 13 | フロントエンド全画面の実装 | 画面5+ |
